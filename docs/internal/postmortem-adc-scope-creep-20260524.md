# 验尸报告：ADC 平台功能范围越界

**日期**：2026-05-24
**严重程度**：P2（产品架构）

## 现象

ADC 平台（agent-dev-center）本应是**需求发布和开发管理平台**，类似猪八戒网。但它混入了大量本应独立的功能：

| 不应在 ADC 中的功能 | 应该在哪 | 影响 |
|---------------------|---------|------|
| Agent OKR / Goals | 独立 OKR 服务 | 后端耦合到 ADC 的 DB 和权限系统 |
| Identity 身份管理 | 独立 SSO/身份服务 | 身份和需求混在一起 |
| Family / Health Records | 独立应用 | 个人数据在研发平台上 |
| Profile 个人属性 | 独立服务 | 用户画像和需求系统耦合 |
| Marketplace | 独立应用 | 功能越来越多 |
| Skill Tree | 独立应用 | 技能图谱和需求系统无关 |
| Kanban Board | 独立看板工具 | 需求平台不需要额外的看板 |
| Roadmap | 独立规划工具 | 路线图应该独立 |
| SSO Portal | 独立 SSO 站点 | 统一认证入口应该单独 |

## 根因分析

**为什么会出现这个情况？**

1. 因为没有独立的"产品需求分类"流程，所有新需求都被提交到 ADC，开发图方便就直接在 ADC 里加模块
2. 因为 ADC 已经有用户认证和数据库，新功能直接复用最省事
3. 因为没有一个产品负责人判断"这个功能应该放在哪个平台"
4. 因为最初 P0 急于交付，功能先加再说，没有做架构分层

## 为什么现有流程没拦住

- AGENTS.md 只规定了一个项目 = 一个 Repo 的结构约束
- 没有产品层面的"哪些功能属于哪个平台"的划分规则
- PM 审核流程刚建立（2026-05-18），还没覆盖到平台边界审查
- ADC 的架构设计没有考虑多平台拆分

## 提取的长期原则

### 原则 1：ADC 平台的职责边界
- **ADC（Agent Dev Center）= 需求发布 + 开发管理 + 交付验收**
- 不属于这三样的功能，不能加在 ADC 中
- 例外：老板亲口说明要改造 ADC（eg. "把 ADC 改造成统一平台"）

### 原则 2：新功能必须先回答"放在哪个平台"
- 每个新需求提交时，必须先明确所属平台
- 如果现有平台都不适合，必须创建新平台而不是硬塞到旧平台
- PM 审核时必须检查平台归属

### 原则 3：混入的功能需要逐步剥离
- 已经混入的功能不能直接删除（数据迁移成本）
- 但必须标记为"待剥离"，新版本逐步拆出独立服务
- 新增功能不再往 ADC 里加

## 修复方案

1. ✅ 验尸 + 落盘原则（本次）
2. 🔲 在 AGENTS.md / docs/postmortem-lessons.md 记录 ADC 平台职责边界
3. 🔲 把已混入的功能列一个剥离计划
4. 🔲 新功能通过 PM 审核时，必须确认平台归属
