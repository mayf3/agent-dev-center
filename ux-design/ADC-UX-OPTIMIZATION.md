# ADC 前端用户体验优化设计

> 项目：Agent Dev Center 前端 | 需求编号：20d532c0 | 优先级：P3
> 设计范围：提交流程优化、列表页信息架构、移动端适配、设计语言与组件规范

---

## 一、当前用户体验评估

### 1.1 现有限定

经代码审查，当前 ADC 采用 **React + TypeScript + Ant Design 5**，使用 `cssVar` 主题变量，路由基于 `react-router-dom` V6+，后端基于 Express + Prisma。

### 1.2 评估摘要

| 维度 | 现状 | 问题等级 |
|------|------|----------|
| 需求提交流程 | 7 字段表单，1 步提交 | ⚠️ 中等 |
| 列表页 IA | 表格+筛选，固定列宽 | ⚠️ 中等 |
| 移动端适配 | 响应式 CSS，底部导航 4 项 | 🔴 高 |
| 设计规范 | Ant Design 原生主题，未建 Token 体系 | ⚠️ 中等 |
| 导航 IA | 桌上 9 项+移动 4 项，层次不清 | 🟡 低 |
| 无障碍 | 无特别处理 | 🟡 低 |

## 二、设计原则

本方案遵循 SOUL.md 中定义的设计原则：

| 原则 | 在本项目中的应用 |
|------|-----------------|
| 用户优先 | 区分三类用户：提交者(requester)、开发者(developer)、管理员(admin)，各角色看到不同界面 |
| 简洁至上 | 需求提交流程从 1 步拆为 2 步引导式，降低心智负担 |
| Fitts 定律 | 所有触控目标 ≥ 44pt，常用操作放在大拇指自然覆盖区域（移动端底部） |
| Hick 定律 | 列表页默认展示关键字段，次要信息通过展开/悬浮查看，减少决策密度 |
| 格式塔接近性 | 相关表单字段分组展示，视觉间距区分 |
| 一致性 | 所有列表使用统一卡片/表格模板，所有操作按钮位置统一 |
| 渐进式展示 | 提交流程先展示必填，再展开可选；详情页 Tab 切换 |
| 移动优先 | 640/960/1280 断点，先设计移动端体验 |

## 三、设计系统与组件规范

### 3.1 Design Token 三层体系

```
全局 Token ──→ 别名 Token ──→ 组件 Token
```

**层级 1：全局 Token**（所有组件共享的基础值）

```json
{
  "color": {
    "primary": "#1677ff",
    "success": "#52c41a",
    "warning": "#faad14",
    "danger": "#ff4d4f",
    "info": "#1677ff",
    "bg": {
      "layout": "#f5f7fb",
      "container": "#ffffff",
      "elevated": "#ffffff"
    },
    "text": {
      "primary": "rgba(0,0,0,0.88)",
      "secondary": "rgba(0,0,0,0.45)",
      "disabled": "rgba(0,0,0,0.25)"
    },
    "border": "#f0f0f0"
  },
  "radius": {
    "base": 8,
    "sm": 6,
    "lg": 12
  },
  "font": {
    "family": "-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif",
    "size": {
      "sm": 12,
      "base": 14,
      "md": 16,
      "lg": 20,
      "xl": 24
    }
  },
  "spacing": {
    "xs": 4, "sm": 8, "base": 12, "md": 16, "lg": 24, "xl": 32
  }
}
```

**层级 2：别名 Token**（语义化映射，支持暗色模式）

```
--color-bg-subtle      = color.bg.layout （灰色背景区域）
--color-bg-card         = color.bg.container （卡片背景）
--color-border-default  = color.border （默认边框）
--text-body             = color.text.primary （正文）
--text-caption          = color.text.secondary （辅助文字）
--touch-target-min      = 44px （Fitts 验证）
```

**层级 3：组件 Token**（每个组件的命名变量）

如 `--button-primary-bg`、`--card-header-bg`、`--input-border-focus` 等，在下方组件规范中定义。

### 3.2 核心 10 组件规范

#### 1. Button（按钮）

| 变体 | 用途 | 使用场景 |
|------|------|---------|
| `primary` | 主要操作 | 提交需求、通过审批 |
| `default` | 次要操作 | 取消、返回 |
| `text` | 最轻量 | 查看详情、编辑 |
| `danger` | 危险确认 | 拒绝需求、删除 |
| `link` | 导航 | 跳转链接 |

**交互规范**：
- 触控目标 ≥ 44×44pt（移动端按钮统一高度 44px）
- 加载态显示 spinner，禁用交互
- 主操作按钮始终在左侧（移动端底部操作栏主按钮靠右）

#### 2. Input（输入框）

- 单行：高度 36px（桌面）/ 44px（移动）
- 多行：默认 4 行，可拖拽调整
- 有计数（showCount）：标题 max=120，描述 max=2000
- 占位符需带有示例（如"例如：搭建需求评审自动化流程"）

#### 3. Checkbox / Radio

- 点击区域 ≥ 44×44pt（实际 ✓ 框 18×18，周围 padding 13×13）
- 多选用 Checkbox，单选用 Radio，二选一用 Toggle

#### 4. Toggle（开关）

- 用于：需求的"是否涉及 UI"标记、"是否公开"标记
- 高度 22px，宽度 40px
- 带有明确的 label 文字，避免单独使用

#### 5. Select（选择器）

- 搜索框高度同 Input（36/44px）
- 下拉选项 5-8 项为理想长度
- 移动端使用 Picker 模式（底部滑出）

#### 6. Card（卡片）

```json
{
  "padding": { "desktop": "16px 20px", "mobile": "12px 14px" },
  "borderRadius": 8,
  "borderColor": "var(--color-border-default)",
  "boxShadow": "none"
}
```

- 卡片标题使用 `space-between` 布局，内容区和操作区分离
- 可点击卡片：hover 轻微上浮（`translateY(-1px)`）
- 移动端卡片间距 10px

#### 7. Modal（弹窗）

- 桌面：居中，最大宽度 720px（详情）/ 480px（确认）
- 移动：全屏 Modal，底部操作栏固定
- 步骤类 Modal：支持多步（Step Modal）

#### 8. Navigation（导航）

- **桌面**：左侧 Sider，宽度 220px，9 个菜单项分组为 3 类
- **移动**：底部 Tab Bar（4 项）+ 左上汉堡菜单（抽屉展开全部）

导航信息架构优化（见第五节）

#### 9. Toast（消息提示）

- 成功：绿色 `message.success`，自动消失（3s）
- 错误：红色 `message.error`，持久显示直至手动关闭
- 警告：黄色 `message.warning`，自动消失（5s）
- 加载：`message.loading`，操作完成后自动替换

#### 10. Data Table（数据表格）

- 桌面：完整表格，前 3 列固定
- 移动：卡片列表，每行 1 个需求卡片
- 空状态：显示"暂无数据"插图 + 操作引导

## 四、需求提交流程优化

### 4.1 问题分析

**现状**：
- 7 个字段挤在一屏，完整滚动才能填完
- `assignee` 字段暴露给提交者，但提交者无权设置
- "附件链接"作为 URL 输入框，不符合用户心智模型
- 缺乏填写引导，用户常写不完整

**优化目标**：
- 减少表单填写焦虑
- 区分必填/选填
- 增加引导文案和示例
- 提交后即时反馈

### 4.2 方案：2 步渐进式引导

#### 第 1 步：快速提交（必填）

```
┌──────────────────────────────────────────┐
│ 📝 提交需求                              │
│                                           │
│ 需求标题 *                                │
│ ┌──────────────────────────────────────┐ │
│ │ 例如：搭建需求评审自动化流程         │ │
│ └──────────────────────────────────────┘ │
│                                           │
│ 需求描述 *                                │
│ ┌──────────────────────────────────────┐  │
│ │ 支持 Markdown。推荐写明：             │  │
│ │                                     │  │
│ │ 1. 背景：为什么需要这个需求          │  │
│ │ 2. 目标：期望实现什么效果            │  │
│ │ 3. 验收标准：如何判断完成            │  │
│ └──────────────────────────────────────┘ │
│                                           │
│ 优先级 *    [ P2 ▼ ]                      │
│                                           │
│ ┌────────────────────────┐ ┌──────────┐  │
│ │ 取消                  │ │ 提交需求 │  │
│ └────────────────────────┘ └──────────┘  │
│                                           │
│ ─── ─── ─── ─── ─── ─── ─── ─── ───      │
│ (点击展开更多选项)                         │
└──────────────────────────────────────────┘
```

**交互说明**：
1. 默认只展示 3 个必填字段（标题、描述、优先级）
2. "更多选项"折叠区域包含 department、dueDate、attachment
3. `assignee` 字段移除——提交者无权分配负责人
4. attachment 改为文件上传按钮（多附件支持）

#### 第 2 步：成功引导（提交后）

```
┌──────────────────────────────────────────┐
│ ✅ 需求已提交！                           │
│                                           │
│ 你的需求「需求标题」已进入审核流程        │
│                                           │
│ 接下来会发生：                            │
│   1. PM / CTO 审核需求内容                │
│   2. 审核通过后分配开发负责人              │
│   3. 开发完成后进入测试/验收                │
│                                           │
│ 📊 查看需求状态 ──→                       │
│ 📋 返回需求列表 ──→                       │
│ ➕ 继续提交                               │
│                                           │
│ ─── ─── ─── ─── ─── ─── ───              │
│ 💡 小提示：需求越详细，审核越快通过        │
└──────────────────────────────────────────┘
```

**交互说明**：
1. 提交成功后，用成功状态替代表单
2. 3 个操作入口：查看详情 / 返回列表 / 继续提交
3. 底部有轻量提示，引导用户填写更详细的需求

### 4.3 提交表单组件细节

```
┌─ 附件选择（替换原 URL 输入） ────────────────────┐
│                                                  │
│  附件（可选）                                     │
│  ┌──────────────────────────────────────────────┐│
│  │ 📎 点击上传文件（支持 PDF、图片、文档）       ││
│  │ 或粘贴链接 ████████████████████████████████ ││
│  └──────────────────────────────────────────────┘│
│                                                  │
│  已上传：                                        │
│  📄 demo_spec.pdf           2.3MB      ✕         │
│  📄 screenshot.png          1.1MB      ✕         │
│  📎 https://example.com      链接       ✕         │
└──────────────────────────────────────────────────┘
```

## 五、列表页信息架构优化

### 5.1 问题分析

**现状**：
- 桌面表格 9+ 列，水平滚动
- 筛选栏 5 个条件，信息过载
- 3 个 Tab 中的"我的任务"和"我提交的"容易混淆
- 缺少"近期动态"视图

## 六、移动端适配方案

### 6.1 全局移动端体验

#### 导航结构

现状：底部 4 项 + 汉堡菜单 | 优化：底部 5 项 Tab，内容重排

```
┌──────────────────────────────────┐
│           [状态栏]                │
│  🛠️ Dev Center          ☰  👤  │  ← 顶部 Header
├──────────────────────────────────┤
│                                  │
│         内容区域                   │
│    (滚动)                         │
│                                  │
├──────────────────────────────────┤
│  📊  💡  📋  🔧  ⚙️            │  ← 底部导航 5 项
│ 仪表盘 需求 看板 集市  更多      │
└──────────────────────────────────┘
```

| Tab | 页面 | 说明 |
|-----|------|------|
| 📊 仪表盘 | 当前 Dashboard | 统计概览 |
| 💡 需求 | 需求列表 | 当前页 |
| 📋 看板 | 开发看板 | Kanban 看板，只读模式 |
| 🔧 集市 | Marketplace | 能力集市 |
| ⚙️ 更多 | 弹出面板 | 团队看板、验尸报告、服务监控、改密、退出 |

**Fitts 定律分析**：底部 Tab Bar 固定，拇指自然覆盖所有 5 个按钮（移动端握持时拇指最易触达屏幕底部区域）

#### 拖拽优化

- 看板页在移动端改为横向滚动列 + 竖向卡片列表（取消拖拽，改为点击选择状态）
- 减少移动端"拖拽"操作，替换为 Select 下拉选择状态（Hick 定律——移动端拖拽误触率高）

#### 详情页移动端优化

```
┌──────────────────────────────────┐
│ ← 返回  [P2] [开发中]  [⋯]  │  ← 右上角更多操作
│                                   │

│ ┌─ 需求信息 ──────────────────┐ │
│ │ 截止：06-15                 │ │
│ │ 负责人：后端开发工程师      │ │
│ │ ───────────────              │ │
│ │ 描述：点击展开              │ │  ← 折叠长描述
│ └──────────────────────────────┘ │
│                                   │
│ ┌─ 任务进度 ──────────────────┐  │
│ │ ████████░░ 80%             │ │
│ │ 3/4 任务完成                │ │
│ └──────────────────────────────┘ │
│                                   │
│ Tab: [需求描述] [任务] [报告]     │  ← 横向 Tab
│                                   │
└──────────────────────────────────┘
```

### 6.2 断点规则

```
640px  ── 移动端（手机竖屏）
960px  ── 小平板/手机横屏
1280px ── 桌面窄屏
1600px ── 宽屏
```

断点采用内容驱动，不绑定具体设备型号。
组件层优先使用 `Container Queries` 而非 `Media Queries`，确保组件在任何 Layout 中都能自适应。

### 6.3 移动端组件适配表

| 组件 | 桌面端 | 移动端 |

## 八、实施计划

### 8.1 优先级与分期

| 阶段 | 内容 | 工作量预估 | 影响面 |
|------|------|-----------|--------|
| **Phase 1** (P3) | 设计 Token 体系 + 核心组件规范文档 | 2h | 全局 |
| **Phase 2** (P3) | 需求提交流程 2 步式优化 | 4h | 提交流程 |
| **Phase 3** (P3) | 列表页卡片式重构 | 6h | 列表页 |
| **Phase 4** (P3) | 移动端体验全面提升（导航重排 + 组件适配） | 8h | 移动端 |
| **Phase 5** (P3) | 统一设计语言落地（微调现有页面样式） | 4h | 全局 |

### 8.2 实施建议

1. **先做 Phase 1**：建立 token 和组件规范，后续改造有统一基准
2. **Phase 2 和 Phase 3 可以并行**：提交流程和列表页相互独立
3. **Phase 4 依赖 Phase 1**：需要 Token 体系支撑移动端组件样式
4. **所有改动用 Feature Branch**：每个 Phase 独立分支，Code Review

### 8.3 风险提示

- 卡片式列表在数据量大时（500+）需 Virtual Scroll 优化
- 移动端全屏 Modal 需注意键盘遮挡问题（IOS Safari）
- 附件上传需后端支持文件存储，当前仅有 URL 字段
