# ADC Migration Baseline

## Purpose

This baseline allows provisioning a fresh ADC database without replaying the
full 38-migration history, which has ordering issues on fresh databases.

## Cutoff

The baseline covers ALL migrations up to and including:

```text
20260629000000_add_lock_token
```

Migration `20260629000000_add_lock_token` is the **cutoff** — it is the last
migration included in this baseline.  After applying the baseline, the script
registers each covered migration via `prisma migrate resolve --applied`.

## Contents

| File | Description |
|------|-------------|
| `baseline.sql` | Full schema at cutoff (generated by `prisma migrate diff --from-empty --to-schema-datamodel` plus seed/backfill SQL from historical migrations) |
| `covered-migrations.txt` | Fixed list of migration names covered by the baseline |

## Generated vs Added SQL

`baseline.sql` consists of two sections:

1. **Generated** (lines 1-N): Output from `prisma migrate diff --from-empty --to-schema-datamodel=backend/prisma/schema.prisma`.  Includes CREATE TYPE, CREATE TABLE, CREATE INDEX, and ALTER TABLE ADD CONSTRAINT for all Prisma-managed objects.

2. **Added** (after the diff): Seed data and data backfills from historical migration files that are needed for a fresh database but not captured by `migrate diff`.  These are explicitly listed below with their source migrations.

### Seed Data

- `password_policies` default row (from `20260602000100_add_password_policy`)
- `workflow_templates` seed rows with corrected `updatedAt` column (from `20260530150200_add_workflow_engine`, adjusted for NOT NULL constraint)
- `test_env_lock` singleton row (from `20260612190001_seed_test_env_lock`)

### Data Backfills

- `users.password_changed_at` backfill (from `20260602000100_add_password_policy`)

## Regeneration

To regenerate the generated portion:

```bash
prisma migrate diff --from-empty --to-schema-datamodel=backend/prisma/schema.prisma
```

The added seed/backfill SQL must be manually re-applied after regeneration.
Always verify the full baseline.sql against a fresh database before committing.

## Verification

The bootstrap script verifies:
- Database is empty
- `_prisma_migrations` does not exist
- `baseline.sql` SHA-256 matches the recorded manifest
- `covered-migrations.txt` SHA-256 matches
- `DATABASE_URL` does not look like a production connection
